# Session: 2026-04-14

## Goal
Diagnose and fix the 3–5 second cold-launch delay on M4 Pro MacBook Pro, then ship a 1.0.2 release.

## Progress

### Root-cause analysis
- User reported consistent 3–5s cold-launch delay on M4 Pro — unusually slow for a lightweight SwiftUI app on fast hardware.
- Traced launch path: `NetworkQualityApp` → `ContentView` (`@StateObject NetworkQualityViewModel()`) → `NetworkQualityViewModel.init()` → `loadInterfaces()` → `service.getAvailableInterfaces()`.
- Found the smoking gun in `Services/NetworkQualityService.swift:44-56`: a synchronous wrapper used `DispatchSemaphore.wait(timeout: 5)` around a spawned `Task { await AppleNetworkQualityRunner.getAvailableInterfaces() }`.
- The `Task` inherited `@MainActor` isolation from the enclosing class, so it couldn't make progress while the main thread was blocked by `semaphore.wait`. Every launch hit the 5-second timeout and returned the hardcoded fallback `["en0", "en1", "pdp_ip0"]`.

### Research + design
- Confirmed via Apple docs and Swift forums that this is a well-known antipattern: `DispatchSemaphore.wait` from `@MainActor` code is a deadlock trap. The idiomatic fix is SwiftUI's `.task` modifier, which participates in structured concurrency and auto-cancels on view disappear.
- Chose **Option A**: remove the sync wrapper entirely, load interfaces lazily via `.task` on the only consumer (`SettingsView`). Rejected Option B (keep sync API, fix internals) because the interface list is only read by Settings — zero reason to block launch for it.

### Implementation (wave-based `/execute`)
- **Wave 1** (developer agent): three-file refactor
  - `Services/NetworkQualityService.swift`: deleted sync wrapper + `getAvailableInterfacesAsync`; collapsed to single `func getAvailableInterfaces() async -> [String]`.
  - `ViewModels/NetworkQualityViewModel.swift`: removed `loadInterfaces()` call from `init()`; made `loadInterfaces()` async.
  - `ContentView.swift`: attached `.task { await viewModel.loadInterfaces() }` to the `case 3: SettingsView(...)` branch.
- **Wave 2** (verification): clean `xcodebuild` + 3 cold-launch timing samples.
  - Result: **210 / 185 / 197 ms → 197ms avg.** Target was <500ms. **~25× speedup.**

## Decisions
- **Lazy interface loading via `.task` on SettingsView** (not `.task` on root view). Since `availableInterfaces` is only read inside `SettingsView` (verified via grep — sole consumer at `Views/SettingsView.swift:83`), attaching `.task` there means launch pays zero cost and the user only pays the `ifconfig -l` roundtrip when actually opening Settings. Structured concurrency handles cancellation automatically if Settings closes mid-load.
- **Don't notarize the DMG itself.** The app inside is notarized and stapled, and the v1.0.1 DMG was not notarized either — matching prior practice keeps the download experience consistent.
- **Exclude `APP/` and `02_Design/` from git.** Release artifacts and large design assets bloat the repo without serving source-control purposes; release DMGs live on GitHub Releases instead.
